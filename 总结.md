# Scrapy 使用水平评价

基于对整个 CnBlog 项目的深入分析，以下是当前 Scrapy 使用程度的评价。

---

## 综合评级：中高级（进阶使用者）

已经超越了入门阶段，对 Scrapy 的核心机制有较深理解，并能组合多个高级特性构建一个接近生产级的爬虫系统。项目同时提供了两种 Spider 实现（手动分页 + CrawlSpider 声明式规则），展示了对不同爬取策略的掌握。

---

## 做得好的部分（亮点）

| 模块 | 评价 |
|---|---|
| **全异步架构** | 使用 `AsyncioSelectorReactor` + async 回调 + async Pipeline，这是 Scrapy 2.x 的现代用法，很多人还没切到这个模式 |
| **Pipeline 链设计** | 5 个 Pipeline 分工明确（清洗→校验→图片下载→透传→批量入库），优先级编排合理，体现了对 Item Pipeline 执行顺序的理解 |
| **批量数据库写入** | `AsyncBatchDataBasePipeline` 动态构建多行 INSERT SQL + UPSERT 去重 + 连接池 + 优雅关闭，这是工程化思维 |
| **ImagesPipeline 定制** | 重写 `file_path()`、`get_media_requests()`、`item_completed()` 三个方法，处理了 Twisted Failure，属于中高级用法 |
| **ItemLoader + Processor** | `ArticleLoader` 使用 `MapCompose`/`TakeFirst` 实现提取逻辑与清洗逻辑解耦，`cover_image`/`created_at`/`views` 等字段配置了独立 Processor |
| **CrawlSpider + LinkExtractor** | `ArticleCrawlSpider` 用声明式 Rule 处理分页，`restrict_css` 精确定位翻页区域，`parse_list` + `parse_detail` 两阶段提取设计合理 |
| **Cookie 池管理** | Redis 同步/异步双实现（`CookieManager` + `AsyncCookieManager`），SCAN 迭代、生成器模式、按账号管理，有生产环境意识 |
| **反检测登录** | Playwright 自动化登录 + webdriver 属性覆盖 + 浏览器指纹伪装 + reCAPTCHA 检测 |
| **爬虫礼仪** | AutoThrottle、随机延迟、robots.txt 遵守、域名级并发控制 |
| **自定义 Extension** | 通过信号监听实现统计汇总（`item_scraped`/`item_dropped`/`spider_closed`），理解了 Scrapy 的信号机制和扩展点 |
| **双 Spider 实现** | `ArticleSpider`（手动分页）和 `ArticleCrawlSpider`（CrawlSpider）并存，展示了两种爬取策略的对比实现 |

## 还有提升空间的部分

| 模块 | 现状 | 建议 |
|---|---|---|
| **RandomUserAgentMiddleware** | 已实现（26 条 UA），但 settings 中注释掉了，未启用 | 取消注释启用，配合 Playwright 的浏览器指纹效果更好 |
| **ProxyMiddleware** | 已写好框架但注释掉了 | 生产环境需要代理轮换，建议实现并启用 |
| **Cookie 刷新** | Cookie 在类定义时一次性获取，爬取过程中不会轮换或刷新 | 长时间爬取可能导致认证失效，建议实现定时刷新或信号驱动的 Cookie 轮换 |
| **AsyncCookieManager 未使用** | 异步版本已实现，但两个 Spider 都用的同步 `CookieManager` | 在全异步架构中应切换到 `AsyncCookieManager`，避免阻塞事件循环 |
| **Spider 拆分** | `ArticleSpider` 同时处理 `www` 和 `news` 两个子站，通过 `if/else` 分支路由 | 建议拆成两个 Spider，各自职责更清晰（`ArticleCrawlSpider` 目前也是单文件处理两个子站） |
| **错误处理** | `parse_detail` 中 `except Exception: pass` 会静默吞掉错误 | 建议至少记录日志（当前已有 `logger.error`，但 `pass` 后 item 丢失无感知） |
| **CleanPipeline 逻辑** | `value.strip("")` 只去除空字符串参数，不去除空白字符 | 应改为 `value.strip()` 才能真正去除首尾空白 |
| **CnblogItem 未清理** | `items.py` 中 `CnblogItem` 是空壳，从未使用 | 删除无用代码，减少混淆 |
| **OffsiteMiddleware 被禁用** | settings 中设为 `None` | 如果 `allowed_domains` 已正确配置，应启用此中间件防止请求到站外 |
| **Feed Exports** | 基本没用 Scrapy 内置的导出功能（JSON/CSV），而是自己写了 `ArticleJsonWritePipeline` | 内置 Feed Exports 支持更多格式和存储后端，且已配置了 `FEED_EXPORT_ENCODING` |
| **工具函数命名** | `str_datetime(str)` 参数名 `str` 遮蔽了 Python 内置类型 | 改为 `date_str` 或 `text` |
| **`get_random_cookie()` 路径** | `utils/common.py` 中引用 `scripts/cookies.js`，实际文件在 `core/cookies.js` | 路径不一致，应修正 |

## 两种 Spider 实现对比

| 维度 | ArticleSpider | ArticleCrawlSpider |
|---|---|---|
| **基类** | `scrapy.Spider` | `CrawlSpider` |
| **分页方式** | 手动 XPath 提取下一页链接 | `Rule` + `LinkExtractor` 声明式 |
| **数据提取** | 直接 `item['field'] = value` 赋值 | `ArticleLoader` + Processor |
| **代码量** | 较多（分页逻辑内嵌） | 较少（分页由 Rule 驱动） |
| **灵活性** | 高（完全控制请求流程） | 中（受 Rule 匹配规则约束） |
| **适用场景** | 爬取逻辑特殊、需要动态生成请求 | 规则化的列表页+详情页爬取 |

## 进阶路线建议

### 短期（补齐已有代码的缺口）

1. **启用 RandomUserAgentMiddleware**：取消 settings 中的注释，让 UA 轮换生效
2. **修复 CleanPipeline**：`value.strip("")` → `value.strip()`
3. **切换到 AsyncCookieManager**：在全异步架构中避免同步 Redis 调用阻塞事件循环
4. **修正 `get_random_cookie()` 路径**：`scripts/cookies.js` → `core/cookies.js`
5. **清理无用代码**：删除 `CnblogItem`、`test.py` 中的调试代码

### 中期（生产化加固）

1. **代理轮换**：启用并完善 `ProxyMiddleware`，接入代理池
2. **Cookie 自动刷新**：通过信号监听 `spider_idle` 或定时检查 Cookie 有效性，失效时触发重新登录
3. **Spider 拆分**：将 `www` 和 `news` 子站拆为独立 Spider
4. **启用 OffsiteMiddleware**：配合 `allowed_domains` 防止请求外泄

### 长期（架构升级）

1. **引入 Scrapyd / Scrapy Cloud**：实现爬虫的远程调度和部署
2. **监控告警**：接入 Prometheus/InfluxDB，对爬取速率、错误率、数据库写入量做实时监控
3. **增量爬取**：基于 `updated_at` 时间戳实现增量更新，避免全量扫描
4. **数据质量 Pipeline**：增加内容去重（simhash）、敏感词过滤等后处理环节

---

## 总结

当前项目在异步架构、Pipeline 链、数据库批量写入、图片下载、Cookie 管理、ItemLoader 数据提取、CrawlSpider 声明式爬取等方面已达到中高级水平。两种 Spider 实现（手动 + CrawlSpider）并存，展示了对 Scrapy 不同抽象层级的理解。

主要提升方向从「功能补齐」转向「生产化加固」：启用已实现的 UA 轮换和代理中间件、切换异步 Cookie 管理、修复已知 bug，再配合监控告警和增量爬取策略，即可构成一个完整的生产级爬虫系统。
